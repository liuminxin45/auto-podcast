# PodFlow Studio Constitution

> 本文档定义 PodFlow Studio 的项目定位、核心优势、非目标、取舍原则和长期演进边界。
> 它不是功能清单，而是用于约束 README、Roadmap、Issue、PR、架构设计和后续重构的核心原则文档。

---

## 1. 项目定位

PodFlow Studio 是一个 **本地 AI 播客工作台**，用于将 RSS、网页、热点和手动素材转化为播客脚本、真人录音 / AI 配音音频，并生成可发布的 RSS。

项目的核心不是“采集更多新闻源”，也不是“一键生成一个音频文件”，而是：

> 将播客生产过程拆成可视化、可介入、可恢复、可验证、可发布的工作流。

---

## 2. 一句话定位

推荐对外使用：

> 本地 AI 播客工作台，支持从 RSS、网页和热点素材生成脚本、音频与 RSS。

README 首屏可使用：

> PodFlow Studio 是一个本地 AI 播客工作流系统，可将 RSS、网页、热点和手动素材转为播客脚本、真人录音 / AI 配音音频，并生成可发布的 RSS。

不推荐使用：

> AI 新闻播客生成器。

原因：这个说法过窄，容易与纯 RSS 自动化工具、Podcastfy、NotebookLM 类工具正面竞争，也无法体现本项目的工作流优势。

---

## 3. 核心优势

### 3.1 本地优先

PodFlow Studio 优先作为本地桌面端工具运行。

核心价值：

* 用户素材、脚本、音频、配置和中间产物优先保留在本地；
* 用户可以检查、修改和恢复每个阶段结果；
* 适合个人创作者、开发者和小团队搭建自己的内容生产链路；
* 不强依赖单一 SaaS 平台。

项目应持续强化：

* 本地项目目录；
* 本地状态保存；
* 本地音频产物；
* 本地 RSS 输出；
* 本地可复现 Demo。

---

### 3.2 工作流生产，而不是一次性生成

PodFlow Studio 的核心范式是：

```text
素材接入
→ 素材整理
→ 选题构思
→ 脚本写作
→ 真人录音 / AI 配音
→ 音频后处理
→ 审核
→ RSS 发布
```

项目不应把自己描述成“输入一个链接，生成一个 mp3”的简单工具。

核心优势在于：

* 每个阶段有明确产物；
* 用户可以在关键节点介入；
* 中间结果可以保存和恢复；
* 生成结果可以被审查、编辑、重跑；
* 生产过程可以被工程化验证。

---

### 3.3 Human-in-the-loop

PodFlow Studio 不追求完全无人值守的黑盒生成。

项目应默认承认：

> 播客是内容产品，不是纯技术产物。
> 好的播客需要选题、结构、语气、节奏、事实检查和人工判断。

因此，项目必须保留人工介入点：

* 素材筛选；
* 选题确认；
* 脚本编辑；
* 分段调整；
* 角色 / 声音选择；
* 真人录音替换；
* 音频审核；
* 发布前确认。

不应把“全自动”作为主要卖点，除非在特定场景下能够提供稳定的事实校验、质量评估和发布回滚机制。

---

### 3.4 统一 Workflow State

Workflow State 是项目的核心工程资产。

所有生产阶段应围绕统一状态流转：

```text
输入素材
→ 标准化内容
→ 候选主题
→ 脚本结构
→ 分段脚本
→ 音频片段
→ 后处理结果
→ 发布产物
```

核心原则：

* 每个节点只做明确职责；
* 每个节点输入输出可追踪；
* 状态结构应版本化；
* 状态迁移应可维护；
* 失败节点应可重试；
* 中间产物应可检查。

后续需要补齐：

* `workflow-state.schema.json`
* `docs/workflow-state.md`
* `docs/node-adapter.md`
* state migration 机制
* 示例 state 文件

---

### 3.5 可视化生产台

PodFlow Studio 的相对优势不是 CLI，也不是模型数量，而是可视化生产流程。

桌面端 UI 应突出：

* 当前处于哪个阶段；
* 每个阶段产物是什么；
* 哪些步骤已完成；
* 哪些步骤失败；
* 用户现在可以做什么；
* 哪些内容可以编辑；
* 哪些结果可以回滚或重跑。

UI 不应只作为参数配置面板存在，而应作为“播客生产控制台”存在。

---

### 3.6 中文内容创作友好

PodFlow Studio 应优先服务中文内容场景。

可以强化：

* 中文 RSS / 网页素材；
* 中文热点源；
* 中文脚本结构；
* 中文播客语气；
* 中文 TTS；
* 中文 README、示例和 Demo。

不应在早期为了“国际化叙事”牺牲中文场景深度。

---

### 3.7 工程可验收

项目应保持工程化验收能力。

包括：

* 可运行的 Demo；
* 可复现的示例输入；
* 可检查的中间产物；
* 可播放的最终音频；
* 可验证的 RSS；
* UI 自动化验收；
* 基础单元测试；
* 构建检查。

工程验收是项目优势，但不应压过用户主线。
README 第一屏应先说明用户价值，再在开发者章节说明 CDP、自动化验收和 AI 协作规则。

---

## 4. 应该强化的内容

### 4.1 README 首屏

必须突出：

* 本地 AI 播客工作台；
* 从素材到脚本、音频、RSS；
* 可视化工作流；
* 可人工介入；
* 可运行 Demo；
* 安装包或最短启动路径。

推荐顺序：

```text
项目一句话定位
→ Demo 截图 / GIF / 音频
→ 快速开始
→ 核心能力
→ 示例项目
→ 架构说明
→ 开发者说明
```

---

### 4.2 Demo

必须提供一个完整 Demo：

```text
examples/demo-news/
  input/
    sample-rss.xml
    sources.example.yaml
  output/
    script.md
    episode.mp3
    feed.xml
  README.md
```

Demo 必须回答：

* 输入是什么；
* 中间生成了什么；
* 最终音频是什么；
* RSS 是否可订阅；
* 用户如何复现。

没有 Demo，项目的核心优势很难被外部用户感知。

---

### 4.3 Release

桌面端项目必须提供 Release。

最低要求：

* Windows 安装包；
* 版本号；
* changelog；
* 已知问题；
* 示例配置；
* 可运行 Demo。

没有 Release，项目会被视为源码实验仓库，而不是可用工具。

---

### 4.4 数据源接入协议

项目不需要维护大量新闻源，但必须提供稳定的数据源接入方式。

应支持：

* RSS 配置；
* 网页 URL 配置；
* 手动素材；
* 热点源 adapter；
* 离线 fixture；
* 自定义数据源 adapter。

推荐文档：

```text
docs/data-source-adapter.md
examples/config/sources.example.yaml
examples/fixtures/rss/sample-ai-news.xml
```

核心边界：

> 数据采集覆盖不是项目核心；标准化素材接入是项目核心。

---

### 4.5 RSS 发布能力

RSS 不能只是“生成一个 XML 文件”。

至少应补齐：

* RSS 示例；
* RSS 字段说明；
* feed 校验；
* enclosure 校验；
* 音频 URL 生成规则；
* 本地发布目录结构；
* GitHub Pages / Cloudflare R2 / S3 等发布建议。

---

## 5. 应该弱化的内容

以下内容可以保留，但不应作为 README 主叙事。

### 5.1 AI 编码协作规则

包括：

* `AGENTS.md`
* `ai/rules`
* `ai/knowledge`
* `ai/workflows`
* `.specify`
* `.agents`

这些内容对作者和开发者有价值，但普通用户首先关心的是：

* 能不能运行；
* 能不能生成；
* 生成效果如何；
* 怎么接入自己的素材；
* 怎么发布。

建议放到：

```text
docs/development/ai-collaboration.md
```

或 README 后半部分的“开发者说明”。

---

### 5.2 CDP 自验收

CDP 自验收是工程亮点，但不是普通用户第一感知价值。

应从 README 主线中弱化，放入：

```text
docs/development/acceptance.md
```

README 可保留一句：

> 项目提供 Electron UI 自动化验收，用于验证核心生产链路。

不要在首屏详细展开 CDP 机制。

---

### 5.3 过重的架构解释

Electron、React、Python 节点、IPC、状态协议、runtime setup 等内容应服务于项目理解，而不是喧宾夺主。

README 首屏不应过早出现大量工程内部细节。

建议分层：

```text
README：用户价值 + 快速开始
docs/architecture.md：架构说明
docs/workflow-state.md：状态协议
docs/node-adapter.md：节点开发
docs/development/：开发者流程
```

---

### 5.4 “支持很多输入源”的表达

不应强调“内置很多新闻源”。

应强调：

> 支持标准化接入 RSS、网页、热点和手动素材。

原因：

* 大量数据源会带来维护成本；
* 第三方源稳定性不可控；
* 新闻版权和反爬风险不可控；
* 项目核心不是采集平台。

---

### 5.5 “全自动生成”的表达

应避免把项目描述成完全自动化新闻播客系统。

更稳的表达是：

> 支持自动生成，但默认保留人工审查和编辑节点。

原因：

* 新闻事实风险高；
* AI 容易生成错误、遗漏或过度概括；
* 播客内容需要语气和节奏；
* 发布前应保留人工确认。

---

## 6. 应该删除或迁移的内容

### 6.1 安装阶段的强副作用

不应在 `postinstall` 中做重型网络同步、外部仓库同步、Python editable install 或固定镜像源配置。

应迁移为显式命令：

```bash
npm run setup
npm run setup:python
```

安装原则：

> install 只安装依赖，不做业务初始化。

---

### 6.2 硬编码国内镜像源

不应在默认安装命令中硬编码清华源、公司源或地区性镜像源。

原因：

* 海外用户可能失败；
* CI 环境可能失败；
* 企业网络可能失败；
* 这会降低开源项目通用性。

应改为文档说明：

```bash
pip install -e .
```

如需国内加速，放在 FAQ：

```bash
pip install -e . -i https://pypi.tuna.tsinghua.edu.cn/simple
```

---

### 6.3 需要 fork 才能适配数据源的路径

不应把 fork 作为默认扩展方式。

正确路径：

```text
配置文件
→ adapter
→ 插件
→ fork
```

fork 应该是最后手段，不是推荐路径。

---

### 6.4 过度内部化的文档

README 中不应长期保留大量只对作者本人有用的内部协作规则。

应迁移到：

```text
docs/development/
```

或压缩为开发者入口。

---

### 6.5 未兑现的能力宣传

任何没有 Demo、测试、示例或文档支撑的能力，不应放在 README 首屏。

例如：

* 大规模新闻自动化；
* 高质量播客生成；
* 生产级发布；
* 多平台稳定运行；
* 完整播客运营系统；
* 多模型生态。

没有证据前，可以放 Roadmap，不要放核心能力。

---

## 7. 非目标

PodFlow Studio 明确不做以下事情。

### 7.1 不做大规模新闻采集平台

项目不负责长期维护大量新闻源，也不承诺对所有网站稳定抓取。

项目只负责提供：

* 少量示例源；
* 标准化数据源配置；
* adapter 扩展机制；
* 本地 fixture；
* 清晰的输入格式。

---

### 7.2 不做纯云端 SaaS

项目优先本地运行，不以云端账号体系、多租户、计费、团队空间作为近期目标。

---

### 7.3 不做通用 NotebookLM 替代品

项目不做泛知识库、全文搜索、长期记忆、资料问答平台。

可以支持素材整理，但不应偏离播客生产主线。

---

### 7.4 不做通用 TTS 平台

项目可以接入多个 TTS provider，但不应把声音克隆、声音市场、音色训练作为主线目标。

核心目标是完成播客生产，而不是成为 TTS 基础设施。

---

### 7.5 不做全自动新闻发布机器

项目可以支持自动化生成，但发布前默认应允许人工审核。

尤其是新闻类内容，必须保留事实检查、来源追踪和人工确认空间。

---

## 8. 取舍原则

### 8.1 用户可运行优先于架构完整

当架构完整性和用户可运行性冲突时，优先保证用户能跑通最小闭环。

优先级：

```text
可运行 Demo
> 可听音频
> 可发布 RSS
> 清晰文档
> 架构优雅
```

---

### 8.2 可验证优先于功能数量

新增能力必须尽量配套：

* 示例输入；
* 示例输出；
* 自动化测试；
* 文档说明；
* 失败边界。

没有验证路径的功能，应进入实验区，不应进入主能力列表。

---

### 8.3 配置优先于 fork

用户扩展能力的优先顺序：

```text
配置
→ adapter
→ 插件
→ fork
```

项目应避免让普通用户通过修改源码才能完成常见任务。

---

### 8.4 本地优先，云端可选

默认本地运行、本地保存、本地产出。
云端能力可以作为可选发布方式或模型 provider，不应成为项目基础运行前提。

---

### 8.5 人工可介入优先于黑盒自动化

播客生产中的关键节点应允许人工查看和修改。

不应为了“自动化”牺牲：

* 事实准确性；
* 内容质量；
* 用户控制感；
* 可恢复性；
* 可解释性。

---

### 8.6 少量高质量示例优先于大量不稳定源

项目只需要提供少量示例 RSS / 网页 / 热点数据。

示例必须稳定、可离线、可复现。
不追求内置大量外部新闻源。

---

## 9. README 表达原则

### 9.1 应该突出

README 应优先突出：

* 本地 AI 播客工作台；
* 可视化生产工作流；
* RSS / 网页 / 热点 / 手动素材接入；
* 脚本生成；
* 真人录音 / AI 配音；
* 音频后处理；
* RSS 发布；
* Demo；
* Release；
* 快速开始。

---

### 9.2 应该弱化

README 应弱化：

* 内部 AI 协作规则；
* CDP 实现细节；
* spec-kit 流程；
* 过细的目录说明；
* 大量开发者脚本；
* 未经 Demo 支撑的能力宣传。

---

### 9.3 应该避免

README 应避免：

* 宣称生产级；
* 宣称全自动可靠发布；
* 宣称支持所有新闻源；
* 宣称高质量播客生成但不给样例；
* 把安装过程写得复杂；
* 把 fork 作为默认扩展方式。

---

## 10. Roadmap 原则

Roadmap 应围绕“让核心优势被看见”展开。

优先级建议：

### P0

* Windows Release；
* 完整 Demo；
* 示例 RSS fixture；
* 最短 Quickstart；
* 可播放示例音频；
* 可发布 RSS 示例；
* README 重构。

### P1

* CLI 无头生成；
* workflow state schema；
* 数据源 adapter 文档；
* RSS 校验；
* provider 支持矩阵；
* GitHub Pages / R2 发布说明。

### P2

* Docker 后端生成链路；
* 插件化数据源；
* 多节目管理增强；
* 更丰富 TTS provider；
* 多语言 UI；
* 自动化定时生成。

---

## 11. 质量门槛

任何进入主分支的核心能力，至少满足以下条件之一：

* 有 Demo；
* 有测试；
* 有文档；
* 有示例配置；
* 有可检查产物；
* 有失败边界说明。

对于播客主链路功能，最低要求更高：

```text
输入示例
→ 生成脚本
→ 生成音频
→ 生成 RSS
→ 用户可复现
```

如果做不到，应标记为 experimental。

---

## 12. 项目长期方向

PodFlow Studio 的长期方向不是成为最大的新闻采集器，也不是成为最多模型支持的播客生成库。

更合理的长期方向是：

> 成为一个本地优先、可视化、可扩展、可人工介入的 AI 播客生产工作台。

核心判断标准：

* 用户能否快速跑通一条完整链路；
* 用户能否接入自己的素材；
* 用户能否编辑和审核脚本；
* 用户能否选择真人录音或 AI 配音；
* 用户能否生成可发布 RSS；
* 用户能否理解失败原因并恢复；
* 开发者能否扩展节点和数据源；
* 项目能否用 Demo 和测试证明能力。

---

## 13. 最终原则

PodFlow Studio 应持续坚持：

```text
本地优先
工作流优先
人工可介入
状态可恢复
产物可验证
数据源可配置
发布可闭环
能力不虚标
```

这八条是项目的核心宪法。
